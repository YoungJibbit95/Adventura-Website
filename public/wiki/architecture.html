Architektur

Client/Server von Anfang an

Der Server ist autoritativ. Er besitzt Weltgeneration, Chunk-Mutationen, Entities, Inventarstatus und Simulationstakte. Clients senden Intents und empfangen Chunks, Block-Updates, Entity-Snapshots und Inventar-Snapshots.

Module

  • common: Registries, World/Chunk Data, Packets, Generation, Lighting, Entities.
  • server: Tick Loop, Netty Transport, Auth Hook, World Streaming.
  • client: LWJGL Window, Rendering, UI, Hotbar, Chat, Settings.
  • launcher: Swing Launcher für Singleplayer, Server und Join.
  • tools: Daten- und Asset-Pack-Utilities.

Runtime Shape

Launcher sammelt Startparameter. Client sendet Movement, Block Actions, Craft Requests und Chat. Server validiert und antwortet mit Chunk Data, Accepted Block Updates, Entity Snapshots, Chat Broadcasts und Inventory Snapshots.

Tick Model

Server: feste 20 TPS. Client: variable Render Loop, vorbereitet für Interpolation gegen Server-Snapshots.

Chunk Model

16x16 Footprint, 16x16x16 Sections, Default Dimension Y -64 bis 319, Sky Light und Block Light als separate Werte.

Rendering

LWJGL/OpenGL, Visible-face Meshing, Opaque/Transparent Meshes, Culling, Fog, AO, Soft Shading und animiertes Wasser.

Intent Flow

  1. Client Input: Movement, Block Break/Place, Block Interact, Craft Request und Chat werden als Intents verschickt.
  2. Server Validation: Reichweite, Slot, Cooldowns, Inventar, Blockziel und Station-Nähe werden autoritativ geprüft.
  3. World Mutation: Akzeptierte Änderungen erzeugen Block Updates, Inventory Snapshots, Entity Snapshots oder Chat Broadcasts.
  4. Client Render: Chunks, Lichtwerte, Sprites, HUD und Meshes werden lokal aktualisiert und gerendert.
